# DAY8-API 失败不等于产品全坏

## 内部今日目标

把 VoiceMore 的 AI 能力拆成两个可选择的环节：语音转文字和文字处理，同时学会判断一个 API 错误到底能证明什么、不能证明什么。

## 概念及相关类比

**服务商**像不同快递公司。OpenAI、智谱、DeepSeek 都能帮你处理任务，但每家公司支持的业务不同，有的能听写，有的只能改稿。

**语音转文字**像听写员。你说一段话，听写员负责把声音变成文字。

**文字处理**像编辑。文字已经有了，编辑负责帮你轻微润色、整理格式，但不能把你的语气改没。

**401 错误**像门禁没认出身份证。它说明这次请求的身份认证没通过，但不能证明你没有按门铃，也不能证明电梯、房间、灯全坏了。

**回归验证**像修完车以后重新试刹车。不是说“我觉得修好了”，而是用测试和真实操作确认同类问题再次出现时会报警。

## 一个小动作

今天我做的关键判断是：当 VoiceMore 显示 API 错误时，不能立刻说“产品流程有问题”。

要先问三件事：

1. 失败发生在语音转文字，还是文字处理？
2. 是哪个服务商报错：OpenAI、智谱，还是 DeepSeek？
3. 本地流程有没有单独证据，比如 Alt 触发、录音、界面、构建和自动测试？

这一步让我把“外部服务失败”和“本地产品坏了”分开看。

## 一张知识卡

**是什么：** 多服务商不是简单多放几个 API Key，而是把不同 AI 能力拆开管理：谁负责听写，谁负责润色，谁失败了就说清楚是谁失败。

**什么时候用：** 当一个产品依赖外部 API 时，尤其要这样拆。否则一个 401 就会被误判成整个产品都坏了。

**怎么做：** 先定义服务商能力，再做路由选择。VoiceMore 里，OpenAI 和智谱可以做语音转文字；OpenAI、智谱和 DeepSeek 可以做文字处理；原文模式不需要调用文字处理模型。

**今天最重要的判断：** API 未成功回传信息，不代表 Alt、录音、界面、插入文字这些产品流程也都异常。

## 一个可复用 Prompt

```text
先不要直接修代码。请先帮我判断这个失败发生在哪个环节：本地触发、录音、转写 API、文字处理 API、插入文字、界面反馈，还是 Git/构建环境。请说明这个错误能证明什么、不能证明什么，再给出最小验证步骤。只有在定位清楚后，才允许修改代码。
```

## 今天留下的产品判断

VoiceMore 要给长期办公、写作的人用。过去用户需要在键盘里慢慢打字，现在只需要按 Alt 说话，VoiceMore 自动录音、转文字、轻微润色，并输入到原来的位置。

但越是这样的效率工具，越不能把错误混在一起。一个 API Key 不通过，只说明“身份证没过门禁”；它不能推翻前面已经被验证过的 Alt、录音和本地流程。
