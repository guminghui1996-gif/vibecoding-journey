# DAY6-自动测试通过不等于真实使用正常

## 内部今日目标

分清自动测试和真实操作各自能证明什么，避免看到一片绿色就过早宣布修复完成。

## 概念及相关类比

- **自动测试**：像汽车出厂前在检测台上检查刹车、灯光和仪表，能快速重复检查已经写下来的规则。
- **手工测试**：像真的把车开到路上，检查它与道路、天气和其他车辆配合时会不会出问题。
- **回归测试**：像给发生过的故障装一个报警器，以后同类问题再次出现时立刻提醒。

自动测试和手工测试不是互相替代。前者擅长反复检查程序内部逻辑，后者负责发现程序与真实 Windows 环境配合时的问题。

## 一个小动作

我先看到 19 个自动测试全部通过，又单独运行了 Alt 相关测试，两次都是 8 个通过。但在真实使用时，按完 Alt 后键盘仍然出现失灵。

继续检查后发现，旧测试只确认程序识别到了 Alt，却没有确认 Windows 是否收到了 Alt 松开事件。程序把按下事件交还给了 Windows，却拦住了松开事件，于是系统可能一直以为 Alt 还被按着。

我们先写了一个能够复现问题的测试，确认它会失败；再修复事件传递，完整测试变成 20 个全部通过。最后我重新进行真实操作，键盘表现正常。

## 一张知识卡

**是什么：** 自动测试检查已经写进规则里的行为，手工测试检查产品在真实环境中的完整表现。

**什么时候用：** 涉及全局键盘、麦克风、窗口焦点、剪贴板、托盘和系统权限时，不能只依赖自动测试。

**怎么做：** 先运行自动测试，再按真实使用路径操作；发现问题后稳定复现，补一个会失败的回归测试，完成修复，运行完整测试，最后重复真实操作。

```text
自动测试 → 真实操作 → 复现问题 → 红色回归测试 → 修复 → 全量测试 → 再次真实操作
```

## 一个可复用 Prompt

```text
请不要因为自动测试通过就宣布修复完成。先说明现有测试覆盖了什么、没有覆盖哪些真实系统行为，再给出手工测试清单。若手工测试发现问题，请先补一个能稳定失败的回归测试，再做最小修复，最后提供全量测试和真实操作两类证据。
```
