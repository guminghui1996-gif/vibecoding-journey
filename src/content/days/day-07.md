# DAY7-先让 AI 复述任务，再允许它开工

## 内部今日目标

不用翻资料复述本周关键认知，找出仍然混淆的地方，并把有效经验变成 AI 下次也能遵守的工作规则。

## 概念及相关类比

**Harness** 像装修开工单。它不会代替工人干活，但会在开工前写清楚要改哪里、哪里不能动、怎么验收。AI 能写代码，Harness 负责约束它先把问题想清楚。

今天建立的第一条 Harness 是：先复述目标、范围和验证方法，避免在错误的路上越走越远。

## 一个小动作

我做了一次闭卷复述。过程中发现，我把当前已有的“失败结果窗口”和未来想要的“本地记忆区”混在了一起。这不是答题失败，而是暴露出一个真实产品需求。

我们确认了记忆区的边界：先只保存在本机，不上传云端，不保存录音；保存原始转写、AI 微调结果和用户确认后的最终文字。润色只能轻微调整，不能把个人表达风格磨平。这是一项新需求，目前还没有实现。

随后，我们把 Harness 交给一个没有旧对话背景的新 AI。它只读取规则，就能先复述目标、范围、禁止修改项、风险和验证方法；在没有运行测试时，也没有擅自声称测试通过。这个冷启动结果说明，规则不只在当前聊天里有效。

## 一张知识卡

**是什么：** Harness 是一组约束 AI 工作过程的规则，让正确做法可以被重复执行和检查。

**什么时候用：** 每次准备让 AI 修改代码、排查故障、增加功能或发布版本之前。

**怎么做：** 先让 AI 复述目标、修改范围、禁止修改项、风险和验证方法；确认方向后再开工；结束时要求提供测试、真实操作和版本记录等证据。

```text
复述任务 → 确认边界 → 批准开工 → 执行修改 → 提供证据 → 沉淀规则
```

## 一个可复用 Prompt

```text
先不要修改任何文件。请先用业务语言复述本次目标、修改范围、明确不改的内容、主要风险和验证方法。等我确认后再开始。完成时不要只说“已经修复”，请提供能够证明结果的测试、真实操作和 Git 证据。
```
